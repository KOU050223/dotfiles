## 現在の状態

- Git ステータス: !`git status`
- Diff: !`git diff HEAD`

## コミットの単位についての指針

変更内容を分析し、以下の原則に基づいてコミットを分割するか判断してください：

### 分割すべきケース
- 複数の独立した機能追加が含まれている
- バグ修正と新機能が混在している
- 異なるモジュール/コンポーネントへの変更が含まれている
- リファクタリングと機能追加が混在している
- ドキュメント更新と実装変更が混在している

### 1つにまとめて良いケース
- 1つの機能に関連する複数ファイルの変更
- 同じバグ修正のための複数ファイルの修正
- 関連するテストとその実装コード
- 一貫したリファクタリング作業

### コミット分割の推奨
複数の独立した変更が検出された場合は、以下のように分割することを提案してください：
1. 変更内容をグループ化
2. 各グループごとにコミットを作成
3. 論理的な順序でコミット（例：リファクタリング → 機能追加 → テスト → ドキュメント）

## タスク

1. 上記の変更内容を確認して、コミットの単位を判断
   - 複数の独立した変更がある場合は分割を提案
   - ユーザーに確認を求める

2. 適切なコミットメッセージを作成（Conventional Commits形式を推奨）
   - `feat:` 新機能追加
   - `fix:` バグ修正
   - `docs:` ドキュメント更新
   - `refactor:` リファクタリング
   - `test:` テスト追加・修正
   - `chore:` ビルドプロセスや補助ツールの変更
   - `style:` コードスタイルの変更（フォーマット等）
   - `perf:` パフォーマンス改善

3. コミットを実行（複数に分割する場合は順番に実行）
   - コミットメッセージにAuthor情報やCo-Authored-Byなどの署名は含めない
   - シンプルなコミットメッセージのみを作成

**注:** `~/commit-lint-config.js` が存在する場合は、その設定に従ってコミットメッセージを作成してください（自動チェックはしません）。
